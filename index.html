Self-Hosting Images for Twine

How to make your images available online so they work reliably in your Twine story

Images in Twine need to be reachable by a browser — which means they need to be somewhere on the internet, not just on your computer. This guide walks you through the simplest reliable way to host your own images, and covers some alternatives depending on where your images come from.

→

Why images need to be hosted

A Twine story exports as a single .html file. When you open that file in a browser and it encounters an <img> tag or a background-image CSS rule pointing to a filename on your computer — like "forest.jpg" — the browser cannot load it. The file is on your computer, not on the web. Anyone else who opens your story file sees a broken image.

There are two solutions: put the image somewhere on the internet and reference it with a full https:// URL, or embed the image data directly inside the HTML file using Base64 encoding. This guide focuses on the first approach — hosting — which is more practical for most images and keeps your story file a manageable size.

ℹ️ One exception

If you publish your story to a web server and keep the image files in the same folder as the .html file, relative paths like "images/forest.jpg" work fine. This is covered briefly in Part 3, but for most students a hosted URL is simpler.

Hosting on GitHub

GitHub is primarily a code-hosting platform, but it works just as well for images. A public repository is freely accessible to anyone — including browsers loading your Twine story — and the URLs are permanent as long as you don't rename or delete the file.

Why GitHub specifically

It's free with no storage limits for public repositories within reason. The URLs are stable — they don't expire or change without you doing something. You likely have or will have a GitHub account for other coursework. And the web interface makes uploading a drag-and-drop operation — no command line needed.

Step 1 — Create a repository for your Twine images

If you already have a GitHub account and an existing repository where you want to keep your images, you can skip straight to Step 2. Otherwise:

  1. Go to github.com and sign in (or create a free account if you don't have one yet).

  2. Click the + icon in the top-right corner → New repository.

  3. Give it a name — something like twine-images or comp380-assets.

  4. Set the visibility to Public. This is essential — a private repository cannot serve images to a browser.

  5. Check Add a README file so the repository is not empty.

  6. Click Create repository.

💡 Keep it organised

Create a folder called images inside your repository to keep things tidy. You can do this by clicking Add file → Create new file, typing images/.gitkeep as the filename, and committing. From then on you can upload images directly into that folder.

Step 2 — Upload your image

  1. Navigate to the folder in your repository where you want to store the image (e.g. the images folder).

  2. Click Add file → Upload files.

  3. Drag your image file into the upload area, or click choose your files to browse.

  4. Scroll down and click Commit changes. The default commit message is fine.

  5. Your image is now uploaded and publicly accessible.

⚠️ Filenames matter

GitHub filenames are case-sensitive and spaces in filenames can cause URL problems. Before uploading, rename your file to use only lowercase letters, numbers, and hyphens — for example forest-path.jpg rather than Forest Path.jpg.

Step 3 — Get the raw URL

There are two kinds of URLs for a file on GitHub. You want the raw URL — not the regular page URL.

  1. After uploading, click on your image file in the repository to open it.

  2. Click the Raw button (top right of the file preview area).

  3. Your browser will load just the image. Copy the URL from the address bar.

The raw URL follows this pattern:

https://raw.githubusercontent.com/your-username/your-repo/main/images/forest-path.jpg

You can also construct it manually if you know your username, repository name, branch name, and file path. The branch is almost always main for a newly created repository.

⚠️ Do not use the regular GitHub page URL

The URL you see when browsing GitHub — like github.com/username/repo/blob/main/images/forest.jpg — points to a webpage containing the image preview, not the image itself. Twine will load an HTML page instead of the image. Always use the raw.githubusercontent.com URL.

Step 4 — Use the URL in your Twine story

Once you have the raw URL, you can drop it directly into any of the Harlowe background methods. Here are ready-to-paste code snippets — replace the URL in each with your own.

Method 1 — (enchant:) macro in a header-tagged passage: 

{
(enchant: ?page, (bg:"https://raw.githubusercontent.com/your-username/your-repo/main/images/forest-path.jpg"))
(enchant: ?passage, (text-colour:black) + (bg:#ffffff) + (css:"padding:1.5em"))
}

Method 2 — with passage tags:

tw-story[tags~="forest"] {
  background-image: url('https://raw.githubusercontent.com/your-username/your-repo/main/images/forest-path.jpg');
  background-size: cover;
  background-position: center;
}

tw-passage {
  background: rgba(255, 255, 255, 0.92);
  padding: 1.5em 2em;
  max-width: 700px;
  margin: 2em auto;
}
💡 Test before you submit

After adding the URL to your story, publish it to a file and open that file in a browser. If the background image appears, the URL is working correctly. If it doesn't, double-check you are using the raw URL (starting with raw.githubusercontent.com), not the regular GitHub page URL.

Preparing Your Images

The image itself matters as much as where it is hosted. A poorly prepared image can make your story slow to load, look blurry, or display incorrectly.

File formats

Format Best for Notes
JPG / JPEG Photographs, scenes, atmospheric backgrounds Excellent compression for photos. Does not support transparency.
PNG Graphics, illustrations, anything with transparency Lossless quality. Larger files than JPG for photos.
WebP Either — best of both worlds Newer format with excellent compression. Supported in all modern browsers.
GIF Simple animations Animated GIFs work as backgrounds in Harlowe. Very large file sizes for complex animations.

For background images, JPG or WebP are almost always the right choice. They give you the best quality-to-file-size ratio for photographic content.

File size and compression

Large images slow down your story — especially if players are on a phone or a slow connection. A background image rarely needs to be more than 200–300KB. Here is a rough guide:

Under 200KB
Ideal — loads fast on any connection
200–500KB
Acceptable — noticeable on slow connections
Over 1MB
Too large — compress before uploading

The best free tool for resizing and compressing images is squoosh.app — it runs entirely in your browser, requires no account, and lets you adjust quality interactively while seeing the file size in real time.

  1. Go to squoosh.app and drag your image onto the page.

  2. On the right side, choose MozJPEG (for photos) or WebP as the output format.

  3. Drag the Quality slider down until the file size reaches around 150–250KB. The preview updates live so you can see when quality becomes unacceptable.

  4. Use the Resize option (the expand icon) to reduce the image dimensions too — a background image rarely needs to be wider than 1920px.

  5. Click the download button to save the compressed file, then upload that to GitHub.

Asking Claude for help with image preparation

If you are unsure about file formats, compression settings, or whether a particular image will work as a background, you can ask Claude directly. Some useful prompts:

💬 Example prompts

"I want to use this image as a background in Twine. It's currently 4.2MB as a JPG. What file size and dimensions should I aim for, and what's the best way to compress it?"

"What aspect ratio works best for a full-page background image in Twine on a desktop browser?"

"I have an image with a lot of dark detail. How should I adjust the passage text box so the text is still readable over it?"

"This is my raw.githubusercontent.com URL — can you check it looks correctly formed and tell me what might be wrong if it's not loading?"

💡 You can also share images with Claude

If your AI interface supports image uploads, you can paste or upload an image and ask Claude whether it would work well as a background — whether the contrast is sufficient, whether the composition leaves space for a text box, or how the colours would interact with your story's palette.

Other Hosting Options & Image Sources

Other ways to host images

Option How to get a direct URL Caveats
Flickr Open the photo page → click the download arrow → choose a size → the opened image tab gives you the direct URL. Works reliably for your own uploaded photos. Images must be set to public. Free account has a 1000-photo limit.
Same folder as the .html file Use a relative path like "images/forest.jpg" and keep an images/ subfolder next to the exported .html file. Works when both files stay together. Breaks if the story is shared without the images folder. Not suitable for submitting a single file.
Google Drive Upload file → right-click → Get link → change to Anyone with the link. The sharing URL is not a direct image URL — you need to extract the file ID and construct drive.google.com/uc?id=FILE_ID. Google periodically changes how direct links work. Unreliable for long-term use. Not recommended.
Dropbox Share a file → copy the link → change ?dl=0 at the end to ?raw=1. Works but Dropbox sometimes restricts traffic on free accounts if images are loaded frequently.
Your own website / UNC hosting Upload via FTP or your host's file manager, then use the full https:// URL. Most reliable option if you have web hosting available.

Free image sources

If you need images you do not own yourself, these sources provide free-to-use photography and illustration with stable direct URLs:

Source How to get a URL Licence
Unsplash (unsplash.com) Open any photo → right-click the image → Copy image address. The URL ends in a file format extension. Free for any use including commercial. No attribution required (though appreciated).
Pexels (pexels.com) Same as Unsplash — right-click the image on the photo page → Copy image address. Free for any use. No attribution required.
Wikimedia Commons (commons.wikimedia.org) Open the file description page → click the image to open it full-size → copy the URL from the address bar. Varies by image — check the specific licence on the file page. Most are Creative Commons.
The Public Domain Review (publicdomainreview.org) Right-click images → Copy image address. Or download and host on GitHub. Public domain. Excellent for historical, illustrative, and artistic imagery.
⚠️ Check what you're actually copying

A URL that ends in .jpg, .png, or .webp is a direct image URL. A URL that ends in a page path — like /photos/forest-12345 — is a webpage that contains an image, not the image itself. Open the URL in a new browser tab: if you see only the image with no surrounding interface, it is a direct URL. If you see a full webpage, keep looking.

✓

Checklist

  • Image is in JPG, WebP, or PNG format
  • Image has been compressed to under 300KB using squoosh.app or equivalent
  • Filename uses only lowercase letters, numbers, and hyphens — no spaces
  • GitHub repository is set to Public
  • Image has been uploaded to the repository via Add file → Upload files
  • Raw URL obtained by clicking Raw on the file preview page
  • Raw URL starts with raw.githubusercontent.com — not github.com
  • URL pasted into Twine story (enchant macro or Story > Stylesheet)
  • Story published to file and tested in a browser — background image visible
  • Story file tested on a different computer or sent to someone else — image still loads

Troubleshooting

Problem Most likely cause and fix
Image does not load in the exported story You are using the GitHub page URL (github.com/...) instead of the raw URL (raw.githubusercontent.com/...). Go back to the file on GitHub, click Raw, and copy the URL from the address bar.
Image loads in Twine's Play mode but not in the exported file You may have used a local path like "forest.jpg". Twine's internal preview is more permissive than a browser opening a local file. Replace the path with a full hosted URL.
Image loads for you but not for someone you shared the file with Your GitHub repository may be set to Private. Change it to Public in the repository settings (Settings → Danger Zone → Change repository visibility).
Image URL works in the browser but shows a broken image in Twine The host may be blocking cross-origin requests (CORS). GitHub's raw URLs do not have this problem. If you are using another host, try moving the image to GitHub instead.
Story loads very slowly The image file is too large. Compress it with squoosh.app before uploading. Background images rarely need to be larger than 200–300KB.
Image URL contains spaces or special characters and doesn't work Rename the file before uploading. Spaces in filenames become %20 in URLs and can cause issues. Use hyphens instead: forest-path.jpg not forest path.jpg.
